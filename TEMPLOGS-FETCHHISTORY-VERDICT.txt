VERDICT:  A — NOT VULNERABLE (RLS catches it)

fetchHistory() issues a bare SELECT on temperature_logs with no
application-level org filter. However, Supabase Row Level Security
on temperature_logs enforces org scoping at the database layer via
a facility_id → locations → user_profiles chain. Every
authenticated SELECT is restricted to rows whose facility_id
belongs to a location in the caller's organization.

No cross-org data leakage is possible through this code path.

────────────────────────────────────────────────────────────────

  const fetchHistory = async () => {
    try {
      // Query temperature_logs table
      const { data, error } = await supabase
        .from('temperature_logs')
        .select(`
          id, equipment_id, temperature, temp_pass, corrective_action,
          reading_time, logged_by, input_method, shift, log_type
        `)
        .order('reading_time', { ascending: false })
        .limit(200);
      if (error) throw error;

      const formattedHistory = (data || []).map((entry: Record<string, unknown>) => ({
        id: entry.id as string,
        equipment_id: entry.equipment_id as string,
        temperature: entry.temperature as number,
        temp_pass: entry.temp_pass as boolean | null,
        corrective_action: entry.corrective_action as string | null,
        reading_time: entry.reading_time as string,
        logged_by: entry.logged_by as string,
        input_method: (entry.input_method as string) || 'manual',
        shift: entry.shift as string | null,
        log_type: (entry.log_type as string) || 'hot_holding',
      }));
      setHistory(formattedHistory);
    } catch (err) {
      setPageError(err instanceof Error ? err.message : 'Failed to load temperature history');
    }
  };

OBSERVATION: No .eq() filter on organization_id, facility_id, or
any scoping column. Pure .order().limit(200).

EVIDENCE 5 — RLS SCOPING CHAIN WALKTHROUGH
────────────────────────────────────────────────────────────────

The SELECT policy enforces this chain:

  temperature_logs.facility_id
        ↓ IN
  locations.id
        ↓ JOIN ON
  locations.organization_id = user_profiles.organization_id
        ↓ WHERE
  user_profiles.id = auth.uid()

Translation: A user can only SELECT temperature_logs rows where
the row's facility_id matches a location belonging to the user's
organization. The scoping is transitive — even without an explicit
organization_id column on temperature_logs, org isolation is
enforced through the locations join.

LOCATIONS TABLE RLS: Also ENABLED with full CRUD policies using
the same user_profiles.organization_id pattern. No bypass path.

EVIDENCE 7 — VERDICT RATIONALE
────────────────────────────────────────────────────────────────

The audit flag stated: "TempLogs fetchHistory() has no org_id filter"

This is factually correct at the application layer — fetchHistory()
does not apply any .eq() filter. However, the flag implied a
cross-org vulnerability, which is FALSE.

The database enforces org isolation through RLS:
  1. temperature_logs has RLS ENABLED
  2. The SELECT policy scopes via facility_id → locations → user_profiles
  3. The locations table also has RLS ENABLED with matching policies
  4. There is no bypass path for authenticated users (service_role
     bypass only applies to INSERT, not SELECT)

Additionally, commit 4e2c6a5 documents that a previous attempt to
add an application-level org filter on this table was incorrect
(used non-existent organization_id column) and had to be reverted.
The correct architecture is: RLS handles org scoping via the
facility_id chain.

RECOMMENDATION: No code change needed for fetchHistory(). An
optional defense-in-depth improvement would be to add a
.eq('facility_id', selectedLocationId) filter at the application
layer (using a location ID the user has selected), but this is
a UX optimization (show only one location's logs), not a security
fix. The RLS policy is the authoritative security boundary.
