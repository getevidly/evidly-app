=== INSPECTION 1: GATING FLAG SOURCE ===

Resolver: resolveVisibleSteps() at onboardingChecklistConfig.ts:314-352
Single call site: useOnboardingChecklist.ts:171-174

  const visibleSteps = useMemo(
    () => resolveVisibleSteps(orgIndustry, orgLocationCount, userRole),
    [orgIndustry, orgLocationCount, userRole],
  );

CRITICAL: enrollmentFlags parameter is NOT PASSED. The 4th arg is omitted.
Result: Both sb1383_setup and k12_setup are ALWAYS filtered out.

A12 SB 1383 gating:
  Config line 340: if (step.id === 'sb1383_setup' && !enrollmentFlags?.isSB1383Enrolled) return false;
  Since enrollmentFlags is undefined -> !undefined?.isSB1383Enrolled -> !undefined -> true
  -> Step is ALWAYS filtered out.

  BUG (Appendix 3): The gate is a simple boolean toggle. It does NOT
  check state=CA or generator tier/meals-per-day. A Texas restaurant
  that somehow gets isSB1383Enrolled=true would see this step. The hint
  says "Required for California commercial food generators" but the code
  does not enforce California or generator tier.

A13 K-12 gating:
  Dual gate:
    1. industries: ['k12_school'] -> industryType must be 'k12_school'
    2. Config line 344: if (step.id === 'k12_setup' && !enrollmentFlags?.isK12Enrolled) return false;
  Gate 1 works (industry filter). Gate 2 always filters out (undefined).
  -> K-12 orgs NEVER see this step.

  The K-12 gate correctly checks org type = k12_school via the industry
  filter. The enrollment flag adds a second layer that is non-functional.

Source of other parameters:

  orgLocationCount: from organizations.planned_location_count (hook line 156)
  userRole: from useRole() context (hook line 114)

=== INSPECTION 2: RENDERING BEHAVIOR FOR NON-QUALIFYING ORGS ===

ANSWER: Behavior (a) — steps filtered out entirely.

resolveVisibleSteps() returns a filtered array. Steps that fail any
gate are removed from the array. The rendering component
(OnboardingChecklistCard.tsx) renders only the returned array via
steps.map(). Non-qualifying steps never appear in the UI.

Contrast with dependsOn (sequence gates):
  - Steps with unmet dependsOn ARE rendered but shown LOCKED
  - Lock icon replaces step number
  - Reduced opacity (0.5), cursor-not-allowed
  - Tooltip: "Requires: [Step A], [Step B]"
  - Background: #F8F9FB (light gray), text: #B0B8C5 (muted)

This matches the intended behavior: qualification gates = invisible,
sequence gates = visible but locked.

=== INSPECTION 3: WHERE QUALIFICATION DATA IS CAPTURED ===

SB 1383 qualification data:
  - sb1383_enrolled boolean on organizations table (migration 20260505200000)
  - sb1383_enrolled_at timestamp on organizations table
  - NOT captured during onboarding
  - Only available via Settings > Privacy tab toggle (Settings.tsx:1614-1707)
  - No state=CA check, no generator tier check

K-12 qualification data:
  - k12_enrolled boolean on organizations table (migration 20260505200000)
  - k12_enrolled_at timestamp on organizations table
  - org_type text on organizations table (set to 'k12' on enrollment)
  - NOT captured during onboarding
  - Only available via Settings > Privacy tab toggle (Settings.tsx:1709-1803)
  - industry_type captured in Settings but as uncontrolled input (no Supabase write)

BUG (Appendix 3): Qualification data is NEVER captured during onboarding.
It exists only as a post-onboarding toggle. The conditional onboarding
steps can therefore never appear during first-run onboarding.

BUG (Appendix 3): The enrollment toggles in Settings initialize to false
but never load the org's actual enrollment state from Supabase on mount.
First visit always shows toggles OFF even if org previously enrolled.

=== INSPECTION 4: POST-ONBOARDING ENROLLMENT PATH ===

CONFIRMED: Full post-onboarding enrollment paths exist in Settings.

SB 1383 enrollment (Settings.tsx:1614-1707):
  Route: /settings tab=privacy
  Access: executives, owner_operators only
  Flow: Toggle -> guardAction('sb1383_enrollment') ->
        UPDATE organizations SET sb1383_enrolled=val, sb1383_enrolled_at=now()
        WHERE id=profile.organization_id
  On enable: button links to /sb1383 dashboard

K-12 enrollment (Settings.tsx:1709-1803):
  Route: /settings tab=privacy
  Access: executives, owner_operators only
  Flow: Toggle -> guardAction('k12_enrollment') ->
        UPDATE organizations SET k12_enrolled=val, k12_enrolled_at=now(),
                                  org_type='k12'/'standard'
        WHERE id=profile.organization_id
  On enable: button links to /k12 dashboard

BUG (Appendix 3): Even after enrollment via Settings, the onboarding
checklist hook never passes enrollmentFlags to resolveVisibleSteps().
The conditional steps remain invisible even for enrolled orgs. The
enrollment toggles work (Supabase writes succeed) but the onboarding
checklist never reads them.

=== INSPECTION 5: SCHEMA REQUIREMENT ===

STATED: Tables written by A12 and A13 must exist in PROD regardless of
how many current orgs trigger the conditional. Conditional gates UX
rendering, not schema requirements. When a qualifying org eventually
hits the step, the writes must succeed.

Current state:
  A12 (SB 1383): Writes to food_recovery_records -> TABLE DOES NOT EXIST
  A13 (K-12):    Writes to k12_school_records    -> TABLE DOES NOT EXIST

Both tables have NO migration in the repo. They must be created before
any qualifying org can use these features.

E3: CRITICAL VULNERABILITIES
────────────────────────────────────────────────────────────────────

VULN-001: Dual org-resolution patterns (Pattern A vs Pattern C)
  Pattern A: user_profiles.organization_id -> single org
  Pattern C: user_location_access.user_id -> can return multiple orgs
  If a user has user_location_access rows for org X and Y, Pattern C
  grants access to both while Pattern A only grants access to one.
  Tables using different patterns have different access boundaries.

VULN-002: temperature_logs has no organization_id column
  RLS correctly joins through facility_id -> locations -> user_profiles.
  But client code calling .eq('organization_id', orgId) silently fails
  (returns zero rows, not an error).

VULN-003: organizations INSERT is wide open
  WITH CHECK (true) — any authenticated user can create unlimited orgs.
  Intentional for signup but represents abuse vector.

VULN-004: user_invitations role check mismatch
  RLS checks user_location_access.role IN ('admin','owner').
  App uses user_profiles.role values (owner_operator, platform_admin).
  These are DIFFERENT ROLE SYSTEMS. An owner_operator needs
  user_location_access.role='owner' to manage invitations.

VULN-005: Tables created after RLS catch-all without explicit RLS
  jurisdiction_config_baselines, drift_alert_log, drift_monitor_executions
  (from 20260605000000) created AFTER catch-all (20260531) with NO
  explicit ENABLE ROW LEVEL SECURITY. May be completely unprotected.

VULN-006: vendor_client_relationships — no INSERT/UPDATE/DELETE
  Authenticated users cannot create, modify, or remove vendor
  relationships. useCreateVendor INSERT is silently blocked by RLS.

VULN-007: Missing UPDATE/DELETE on audit-trail tables
  temperature_logs, checklist_completions, checklist_responses,
  cooling_logs — no UPDATE or DELETE policies.
  Possibly intentional (immutable audit trail) but not documented.
